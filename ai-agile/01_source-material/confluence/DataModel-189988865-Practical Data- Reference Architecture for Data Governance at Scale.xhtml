About this Document

This document provides an outline to a proposed reference architecture for implementing data governance at scale for large organizations with highly fragmented business processes and data flows, such as large financial institutions with multiple business lines, individual product-aligned platforms, and centralized functions like finance and risk management. This document forms an introduction to a series of more detailed design documents that cover each component of the architecture in detail which can also be found as links below.

Reference Architecture for Data Governance at Scale

Overview

Assumptions

This reference design for Data Governance at Scale is based on the following underlying input assumptions:

  • Given the scale of potential operations, all components should be able to support “direct” and "federated” implementation approaches

Data Governance Views - Technical vs Business

Data Governance exists at the intersection of technology and business. Technology systems move and transform data around and environment, while business capabilities create data and derive value from its use. As such, it is important that we can represent data through these two lenses. This split personal is represented throughout the architecture and data model.

Layer 1: Business & Data Services

L1

The Business and Data Services layer encompasses all the underlying applications, technologies, and services utilized in an organization's day-to-day data-driven operations. For the purposes of this document, we further categorize these into three categories: Business Operating Systems, Business Reporting & Consolidation Systems, and Reference Data/Master Data Systems.

Business Operating Systems

Business Operating Systems provide the core functional capabilities that enable the day-to-day business operations. This can include the front office (i.e., sales and trading), settlement, marketing, finance, risk management, etc. These systems will tend to create data records themselves as authoritative sources or consume data from others to derive or create additional data.

Business Reporting and Consolidation Systems

Business Reporting and Data Consolidation Systems provide additional data consolidation, aggregation, and analysis capabilities to complement business operating systems. In general, these do not create underlying data but often transform data from multiple sources to meet a specific business output need.

Reference Data / Master Data Systems

Reference Data/Master Data Systems provide an authoritative source for standard reference and other data used to establish common linkages and descriptive data across multiple business lines and products. These services often include key identification, classification, and cross-referencing information.

Layer 2 - Data Governance Functional Capabilities

L2

The Data Governance Functional Capabilities layer provides all the key elements that enable a complete and sustainable data governance capability across an organization at scale. In general, these capabilities all collectively contribute to and leverage data governance metadata in the metadata repository. When operated at scale, many of these capabilities need to be provided through a variety of centralized and federated approaches

.Data Management Portal / End User Experience

The Data Management Portal capability provides a unified interface that enables all business users to interact with the data governance suite of capabilities based on their specific data governance role. In general, front-end capabilities are provided through API integrations to the metadata repository (read) and workflow orchestration capability (create/update/delete).

  • End Users in the Data Management Portal READ data governance data from the metadata repository.

  • End Users in the Data Management Portal CREATE/UPDATE/DELETE data governance data (via orchestration).

API Capability

The API capability provides a set of internal and external methods for retrieving and interacting with data within the metadata manager, based on the current access control roles specified in the roles and responsibilities metadata.

  • The API Framework handles all READ operations from any internal or external capability.

  • API Function handles all CREATE/UPDATE/DELETE operations from the orchestration capability.

Search Capability

The Search capability is an extension of the API services that provides additional high-performance metadata mining and search capabilities, enabling the trawling and retrieval of relevant information from large quantities of data managed within the core metadata management.

  • The Search capability handles all bulk READ operations from internal or external capabilities.

Orchestration Capability

The Orchestration framework enables policy- and rules-driven metadata management capabilities for all data governance activities. All data governance metadata changes should be facilitated through the orchestration framework. The orchestration framework is responsible for conflict resolution, audit trails, and other metadata change management functions.

  • The Orchestration framework receives requests to CREATE/UPDATE/DELETE metadata from other components and implements and performs these actions within the Metadata repository.

Data Lineage Capability

The Data Lineage (Ingestion) capability provides an automated approach for ingesting data related to the configuration of each operating platform and repository, as well as data-at-rest and data-in-transit, to capture the underlying system specifications of data within the current technology environment and how it flows between systems Individual Data Lineage Detectors connect to different platforms to automate the collection of lineage information Further Data Lineage Enrichers can further enrich the captured data based on business rules and other content to enhance the systematically discovered information with additional value-added details (i.e., logical lineage), improving the usefulness of the discovered data Manual data lineage and mapping can also be performed by users (via the Data Management Portal) to enhance the captured data through the Portal further.

  • Data Lineage tools CREATE/UPDATE/DELETE Data Lineage data (via orchestration).

Data Profiling Capability

The Data Profiling (Ingestion) capability enables the ingestion of underlying information about the classification and content of data within the underlying data systems Individual Data Profile Detectors enable the connection to various business systems, allowing for the automatic evaluation and classification of data content Manual data profiling can also be performed by users (via the Data Management Portal) to enhance the captured data through the Portal further.

  • Data Profiling Tools CREATE/UPDATE/DELETE Data Catalog data (via orchestration).

Workflow & Task Management Capability

The Workflow & Task Management Capability enables the assignment and coordinate of work across teams, people, and processes that result from operating data governance at scale Examples of work that needs to be managed around the organisation include the coordination and resolution of Data Definition Gaps, Data Lineage Gaps, Data Quality Issues, TRansaction Monitoring issues etc The Workflow and task management capability leverages the data governance metadata to decide how to assign and coordinate work.

  • Workflow & Task Management capabilities READ from

Data Quality Capability

The Data Quality capability enables the evaluation and recording of the current data quality performance of live data against the latest data quality requirements. Resulting data quality issues are tracked as work items in the work queues. The data quality capability is authoritative for generating and managing Data Quality Issues .

  • The Data Quality capability READs data inventory data from the Data Inventory.

  • The Data Quality capability READs data quality rules from the Data Quality metadata repository.

  • The Data Quality capability executes the quality tests against the live data in business systems OR provides a mechanism for federated data Quality Engines to report Data Quality Issues

  • The Data Quality capability CREATEs/UPDATEs work items (via orchestration).

Data Tracing Capability

The Data Tracing enables the comparison of the current virtual definition of the working environment, as described by the data governance data in the metadata repository, with the true live environment based on the actual data content at rest and in transit. The Data Tracing capability is authoritative for generating and managing Data Tracing Issues.

  • Data Tracing Tools READ from Data Catalog and Data Tracing.

  • Data Tracing Tools READ from Business Operating Systems.

  • Data Tracing Tools CREATE/UPDATE work items (via orchestration).

Transaction Monitoring Capability

Transaction Monitoring enables the analysis of large volumes of transactional data in real-time as it moves through the system, to identify unusual patterns, process breaks, or breaches of established business and compliance rules. The Transaction Monitoring capability is authoritative for generating and managing Transaction Monitoring Issues.

  • Transaction Monitoring Tools READ from Business Operating Platforms and Data Catalog.

  • Transaction Monitoring Tools CREATE/UPDATE alerts and work items (via orchestration).

Reporting and Analytics Capability

The Reporting and Analytics capability enables the in-depth analysis and long-term trending of data governance status across the organisation as a business intelligence capability It leverages a combination of organizational metadata, operating model metadata, data inventory metadata, data quality results data, and others to establish long-term trend metrics across various dimensions Ideally, the reporting and Analytics capability should only be "reporting" on data from other capabilities.

  • Reporting capabilities: READ data from Data Governance Metadata.

Business & Organization Metadata

Business & Organisation metadata describes the organisation’s structure, functions, processes, people, and purposes/outputs. It supports contextual understanding of how data relates to core business operations and strategic outcomes.

Data Policy, Roles, and Operating Model Metadata

Defines the governance framework, including policies, organizational roles, responsibilities, and stewardship models. It also captures workflow and orchestration rules for task routing, approvals, escalations, and SLA tracking, enabling automation across tools and teams.

Technology Systems Metadata

Technology systems metadata describes the systems and applications used by business functions to store, process, and deliver data. It often supports technology dependency mapping and system-to-system lineage (but typically not data-level lineage).

Data Inventory Metadata

See Practical Data: Data Governance At Scale - Data Inventory Capability and Metadata Store

Data Inventory metadata contains the master list of all individual endpoints/data elements across the organisation and additional information, such as name, description, sensitivity (PII vs. Sensitive, etc.), criticality (CDE, etc.), structure (i.e., data hierarchy/scheme), and other descriptive information about the data content (e.g., critical data elements or CDEs) for each data element across the organization Dictionary information can often contain one or more of the business, logical, and physical data inventories.

Data Lineage Metadata

Describes how data is stored, processed, transferred, and transformed from endpoint to endpoint across systems, enabling end-to-end traceability and impact analysis In general, data lineage does not describe the business purpose of the data, but rather how it moves and changes throughout the environment Ideally, data lineage is capable of representing the lowest level of detail, encompassing both external system-to-system endpoint flows and transformations, as well as internal within-application flows/transformations, and then aggregating this information into logical data lineage and system-to-system lineage.

Data Quality Metadata

Defines the required standards for data accuracy, timeliness, and completeness to ensure business processes operate reliably and regulatory expectations are met Ideally, data quality is defined at the intersection between a data element and a purpose—i.e., different data quality standards can apply to the same element depending on the individual purpose.

Layer 3: Operating Model

The Operating Model layer defines the organizational structures, roles, and processes that underpin effective data governance. This layer ensures that governance is not just a set of policies or technologies, but a living, operationalized discipline embedded in day-to-day business activities. Key elements include:

  • Clearly defined roles and responsibilities for data ownership, stewardship, and custodianship.

  • Governance councils and working groups that provide oversight, decision-making, and escalation pathways.

  • Standardized processes for data issue management, change control, and exception handling.

  • Mechanisms for cross-functional collaboration, ensuring alignment between business, technology, and compliance teams.

A mature operating model fosters a culture of accountability and continuous improvement, enabling the organization to adapt to evolving regulatory requirements and business priorities.

Layer 4: Policy

The Policy layer establishes the formal rules, standards, and guidelines that govern the management and use of data across the enterprise. These policies are designed to ensure that data is accurate, secure, and used in a manner consistent with regulatory obligations and organizational values. Key components include:

  • Data classification and handling standards outline the procedures for managing and protecting various types of data.

  • Access and entitlement policies specify who can view, modify, or share data.

  • Data retention and destruction guidelines, ensuring compliance with legal and regulatory mandates.

  • Quality and integrity standards set expectations for data accuracy, completeness, and timeliness.

Effective policy frameworks are regularly reviewed and updated to reflect changes in the regulatory landscape and business strategy. They provide the foundation for consistent, repeatable governance practices across all business units and geographies.

Layer 5: Oversight

The Oversight layer represents the highest level of governance accountability within the organization. This layer is typically embodied by executive leadership, board committees, or designated governance authorities who are responsible for setting the strategic direction for data governance and ensuring that the organization meets its regulatory and ethical obligations. Key responsibilities include:

  • Establishing the vision and objectives for data governance, aligned with the organization's overall strategy.

  • Approving and monitoring the implementation of governance policies, operating models, and technology investments.

  • Overseeing compliance with regulatory requirements and responding to external audits or inquiries.

  • Championing a culture of data-driven decision-making and ethical data use throughout the organization.

Strong oversight ensures that data governance remains a strategic priority, with clear accountability for outcomes and a commitment to continuous improvement.

Data Quality Issues

Data Quality Issues are individual “errors” with the current dataset that need remediation. Data quality issues are detected at a physical level but need to be “understood” at a logical level.

Depending on how DQ issues are generated, a single DQ issue may represent multiple issues across multiple fields on numerous records in a specific system. Usually, this would be a list of errors on a single record or the same mistake on multiple records.

Note: Multiple issues may result (across different systems) from a single root cause, depending on the impact and spread of the “root cause issue”
